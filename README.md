# Barter Box — сервер

Общее веб-приложение для менеджера/логиста, маркетолога и водителей: бартер-проекты,
доставка боксов блогерам, публикации и охваты — всё в одном месте, на одном общем
сервере. Больше не localStorage браузера — теперь все, кто открывает ссылку,
работают с одной и той же базой одновременно.

**Проверено:** 5 000 доставок в базе + 20 одновременных запросов (чтение и запись
вперемешку) — без ошибок и потери данных (`scripts/load-test.js`, см. ниже).

---

## 1. Попробовать локально (2 минуты, для проверки перед деплоем)

Нужен установленный [Node.js](https://nodejs.org) версии 18 или новее.

```bash
cd barter-box-server
npm install
npm start
```

Откройте `http://localhost:3001` в браузере. Готово — там же демо-данные
(4 водителя, 15 доставок, 1 бартер-проект), как и в предыдущей версии.

Чтобы проверить, что всё работает и выдержит нагрузку, не открывая браузер:

```bash
node scripts/load-test.js
```

Скрипт поднимает копию сервера на отдельном порту, прогоняет функциональные
тесты (создание доставки, назначение водителю, доставка с обязательным фото,
импорт CSV и т.д.), затем засеивает 5000 доставок и обстреливает сервер 20
одновременными запросами — и печатает, сколько тестов прошло.

## 2. Как получить рабочую ссылку (https://...) — без своего сервера

Рекомендуемый вариант — **Render.com**, там есть бесплatный тариф и не нужны
знания программирования.

1. Зарегистрируйтесь на [render.com](https://render.com) (можно через Google/GitHub).
2. Загрузите папку `barter-box-server` к себе на GitHub (или попросите
   IT-специалиста сделать это за 5 минут — создать репозиторий и залить файлы).
3. В Render: **New → Web Service** → подключить этот репозиторий.
4. Настройки сборки Render определит автоматически по `package.json`:
   - Build Command: `npm install`
   - Start Command: `npm start`
5. Нажать **Create Web Service**. Через 1–2 минуты Render выдаст рабочую ссылку
   вида `https://barter-box-XXXX.onrender.com` — это и есть ваш адрес, его можно
   сразу рассылать менеджеру, маркетологу и водителям.

**Важный нюанс бесплатного тарифа Render:** на бесплатном плане диск сервера
не сохраняется постоянно — при каждом новом деплое (и иногда при простое) файл
с данными (`data/state.json`) и загруженные фото могут обнулиться до demo-данных.
Это нормально для тестового периода — показать команде, обкатать процесс.

**Когда решите работать по-настоящему** (например, после того как проверите
процесс с командой пару недель) — нужно включить платный план с постоянным
диском (Persistent Disk): это стоит около **$7/мес** за минимальный инстанс
плюс несколько центов за диск. Это тот момент, когда стоит привлечь айтишника
на почасовку — настройка занимает 10–15 минут, но нужно один раз всё сделать
аккуратно (подключить диск к путям `/app/data` и `/app/uploads`).

### Домен
Ссылка вида `barter-box-xxxx.onrender.com`, которую выдаст Render, уже
полностью рабочая и бесплатная — отдельно ничего покупать не нужно. Красивый
собственный домен (`barterbox.kz`) — это отдельная платная покупка (обычно
$10–15/год у любого регистратора), не бесплатная нигде.

## 3. Альтернатива: свой сервер / IT-специалист (Docker)

Если в компании уже есть свой сервер или VPS:

```bash
docker build -t barter-box .
docker run -d -p 3001:3001 \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/uploads:/app/uploads \
  --name barter-box barter-box
```

Флаги `-v` подключают папки `data` и `uploads` с диска сервера внутрь
контейнера — так данные и фото переживают перезапуск и обновление контейнера.
Дальше остаётся направить домен/поддомен компании на этот сервер (это и есть
задача для айтишника — обычно 15–30 минут).

## 4. Что внутри

- `server.js` — весь бэкенд: Express + собственное JSON-хранилище на диске
  (без внешней базы данных — не нужно ничего дополнительно настраивать).
- `public/index.html` — весь фронтенд (одностраничное приложение), обращается
  к серверу через `/api/...`.
- `data/state.json` — сюда пишутся все данные. При первом запуске создаётся
  автоматически с демо-данными.
- `uploads/` — сюда сохраняются фото подтверждения доставки.
- `scripts/load-test.js` — самопроверка (функциональность + нагрузка).

### Роли
- **Менеджер/логист** и **Маркетолог** — вход по логину и паролю (общий на
  роль, как код у водителя). Задаётся переменными окружения на Render, см.
  ниже — без них вход выключен.
- **Водитель** — вход по личному коду (демо: `IVAN-77`, `SERGEY-24`,
  `DANIYAR-31`, `MARAT-08`); коды новым водителям выдаёт менеджер во вкладке
  «Водители».

### Вход по логину: переменные окружения
На Render: Environment → Add Environment Variable.

- `MANAGER_LOGIN`, `MANAGER_PASSWORD`
- `MARKETER_LOGIN`, `MARKETER_PASSWORD`

В код и в публичный репозиторий значения не попадают — только в настройки
хостинга. Пока не заданы — вход для роли выключен (понятная ошибка вместо
угадываемого пароля по умолчанию).

### Инфлюенс интеграции крупные: Telegram-бот по скриншотам
Вкладка «Инфлюенс интеграции крупные» (роль «Маркетолог») считает ROMI по
кривой затухания и позволяет заносить данные вручную или через CSV. Telegram-бот
даёт третий способ: оператор шлёт боту скриншот статистики Instagram-поста с
подписью — логином блогера (`@dana.style`) — бот распознаёт цифры (просмотры,
лайки, комментарии, сохранения, клики) через Anthropic API и сам обновляет
запись в таблице.

Бот работает в том же процессе, что и веб-сервер (long polling) — отдельный
платный сервис на Render не нужен. Включается двумя переменными окружения:

- `TELEGRAM_BOT_TOKEN` — токен от @BotFather (`/newbot` в Telegram, 2 минуты).
- `ANTHROPIC_API_KEY` — ключ с [console.anthropic.com](https://console.anthropic.com)
  (платный API, обычно доли цента за одно фото). Необязательная
  `ANTHROPIC_MODEL`, если понадобится другая модель — по умолчанию
  `claude-3-5-sonnet-20241022`.

Без этих двух переменных бот тихо не запускается, остальное приложение
работает как обычно.

### Ёмкость
Рассчитано и проверено на 5 000 доставок и 20 одновременно работающих
пользователей (см. `scripts/load-test.js`). Если объём вырастет в разы
(десятки тысяч записей, больше 50 человек одновременно) — тогда имеет смысл
переходить с JSON-файла на полноценную базу данных (PostgreSQL), это тоже
задача для айтишника на будущее, не срочная.

### Интеграция с Bitrix / Telegram-ботом водителей
Как и в прошлой версии — кнопка «Отправить в Bitrix» пока симулирует синк
(просто помечает доставку как отправленную). Реальная интеграция с вебхуком
Bitrix и Telegram-ботом водителей (код которого вы получили на сессии) — это
следующий шаг, отдельная задача для айтишника: подключить к этому серверу
исходящий вебхук на `/api/orders/:id/*` события.
